# PRD & DEVELOPMENT SPECIFICATION: AI STORY BUDDY & QUIZ COMPONENT

## TARGET AUDIENCE & DEVICE CONSTRAINTS
- **User Demographics:** Children (Ages 4-9) based in India.
- **Hardware Targets:** Low-to-mid-range legacy Android/iOS devices running with ≈3GB RAM constraints.
- **Performance Threshold:** Fluid UI rendering at a stable ~60 FPS. Code must prioritize a minimal memory footprint, lightweight widget/view tree hierarchies, and aggressive rendering optimization.

---

## 1. TECHNICAL STACK & STATE TOPOLOGY
The component must implement a clean structural separation of layout and business logic using one of the primary framework configurations:

- **Flutter Implementation:** Rigid state separation via Riverpod, BLoC, or clean Provider architectures. Avoid using heavy local inline state mutations (`setState`) for business rules. Use explicit `const` constructor instances and paint boundaries to avoid excessive tree invalidation cycles.
- **Swift Implementation:** Native iOS architecture using SwiftUI. Drive layout states reactively via clean `@StateObject`, `@ObservedObject`, or modern Swift `@Observable` engines. Ensure complete memory safety by preventing leak points inside operational audio block callbacks.

---

## 2. DATA-DRIVEN QUIZ CORE ENGINE
The rendering matrix must completely isolate logic from hardcoded assets. The engine must accept, validate, and cleanly assemble options from a dynamic JSON string schema. It must adapt its visual structure fluidly to handle varying text length or dynamic counts of choices (2, 3, 4, or 5 variants) without layout clipping or button overlaps.

### Baseline Target JSON Payload:
```json
{
  "question": "What colour was Pip the Robot's lost gear?",
  "options": ["Red", "Green", "Blue", "Yellow"],
  "answer": "Blue"
}

Layout Adaptation Matrix Rule-set:
2 Options Supplied: Format horizontally as a clean, split 50/50 block row minus container padding.

3 Options Supplied: Format as a vertical column arrangement or a custom 2+1 staggered block grid.

4 Options Supplied: Format using a symmetrical 2x2 grid layout block.

5+ Options Supplied: Fallback to an optimized scroll list with explicit, secure margin safety wrappers.

3. FINITE STATE MACHINE DESIGN & USER FLOWS
The user workspace acts as a strict state engine, running sequentially across these structural stages without abrupt UI jumps or flashes:

State 1: Idle Presentation State
Present a lightweight visual layout containing a placeholder frame for the AI Buddy Character.

Display a clean text card presenting the explicit narrative string:
"Once upon a time, a clever little robot named Pip lost his shiny blue gear in the Whispering Woods..."

Show a prominent, child-friendly, interactive "Read Me a Story" CTA button. The quiz container remains unrendered or hidden from active view layout logic at this phase.

State 2: Audio Loading / Setup State
Intercept the initial tap action, instantly lock out the button to prevent duplicate triggers, and present an elegant, low-overhead skeleton pulsing loader state.

Spin up the TTS systems or execute API handshakes.

State 3: Active Narration & Speech State
Stream or compute text-to-speech audio outputs using native systems (AVSpeechSynthesizer or flutter_tts), or fallback cleanly to a remote service API (such as an ElevenLabs free tier endpoint).

UX Polish Goal: Provide visual reinforcement by implementing word-by-word active highlighting markers on the text block, or synchronously scale the character's vector frame nodes to simulate speech rhythms.

State 4: Transition State Transition
Upon capture of the audio system's exact execution success/completion callback hook, initiate a transitional sequence.

Cleanly fade/slide up the text panel while pulling the dynamic quiz option matrix into active view via a smooth ease-in-out timeline profile (350ms - 500ms duration).

State 5: Evaluation & Feedback States
Incorrect Choice Selected: Do not lock out selections. Trigger an immediate hardware-level haptic engine kick accompanied by an explicit, rapid horizontal UI shake animation across the choice card. Allow the user to pick an alternate item immediately.

Correct Choice Selected: Lock down all interface input surfaces instantly. Emit an optimized particle/confetti visual animation explosion overlay, swap the buddy vector asset state to a celebrating/joyful expression model, and transition the UI canvas to a satisfying "Success" status presentation.

4. FAULT TOLERANCE & CACHING POLICY
Network Resilience: The application layer must never crash, drop audio buffers, or lock up in an infinite spinning loop when operations drop out. If a connection timeout occurs or a driver throws an alert, catch the failure block gracefully, restore the button state, and surface a warm, intuitive error prompt (e.g., "Pip is clearing his throat! Tap to try again.").

Asset Caching System: When utilizing a remote synthesis pipeline (ElevenLabs), pass strings through a local hashing calculation to serve as unique file IDs. Check for these file IDs in local cache storage prior to executing network requests to minimize bandwidth costs and allow full offline re-plays.

5. RE-ARCHITECTED LOW-RAM RUNTIME OPTIMIZATIONS
Vector Animations: Drive all UI shake routines and systemic confetti fields through native canvas engines or hardware-accelerated drawing pipelines. Avoid heavy multi-frame bitmaps or unoptimized loop logic that could cause garbage collection stutter on 3GB RAM profiles.

Memory Disposal: Prevent explicit memory leaks. Wrap references cleanly using weak reference models ([weak self] in Swift) or execute aggressive, complete garbage sweeps inside termination lifecycles (dispose() tracks in Flutter) to completely sever stream hooks, audio delegates, and animation controller objects.

Model Isolation: Do not calculate or deserialize JSON models inside high-frequency interface paint loops. Perform string parsing once during layout boot initialization to prevent runtime CPU spikes.
